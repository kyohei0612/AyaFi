# ADR-006: note 投稿方式 (案 3 確定の追認)

- **日付**: 2026-04-22
- **ステータス**: 承認済 (2026-04-22、ADR-001 Revisions 2 巡目 §4 と同時確定)
- **決定者**: kyohei
- **関連**: [ADR-001 §Revisions 2 巡目](001-initial-architecture.md), [ADR-009](009-sns-content-generation-engine.md)

---

## Context (問題)

note は公式 API を持たない。自動投稿の手段は 3 通り考えられ、それぞれ規約リスクと
実装工数のトレードオフが異なる。Stage 3 着手前に方式を確定する必要があった。
ADR-001 Revisions 2 巡目で既に議論 + 確定済み。本 ADR はその**追認記録**として独立させ、
Stage 3 実装時に参照しやすくする。

## Decision (結論)

**案 3 採用。アプリはブラウザ自動化を一切行わない。**

### アプリ挙動 (2 機能のみ)

1. **「投稿」ボタン押下時**:
   - Threads / Bluesky に自動投稿
   - **本文をクリップボードにコピー** (note 手動ペースト用)
   - `drafts/` に markdown 自動保存 (ADR-005)
2. **「note で開く」ボタン (投稿ボタン隣に常設)**:
   - Tauri `shell.open("https://note.com/notes/new")` で OS 既定ブラウザ起動
   - 以降の操作は妻が note タブで手動

### 妻の操作フロー

```
[aya-afi UI]              [ブラウザ]
   |
   | 「投稿」押下
   | → Threads/Bluesky 投稿完了 + クリップボードに本文コピー
   |
   | 「note で開く」押下
   |                         | note.com/notes/new が開く
   |                         | 本文エリアに Ctrl+V (貼り付け)
   |                         | タイトル入力
   |                         | 画像を手動アップロード (必要なら)
   |                         | 「公開」押下
```

---

## Alternatives 検討 (ADR-001 と重複、簡略)

| 案 | 自動化レベル | 規約リスク | 実装工数 | 評価 |
|---|---|---|---|---|
| 案 1: playwright で自動入力 | 高 | **中-高** (自動化判定の余地) | 重 | ❌ |
| 案 2: SendInput で Ctrl+V | 中 | 中 | 中 | ❌ |
| **案 3: コピー + ブラウザ起動のみ** | **ゼロ** | **ゼロ** | **最軽** | **✅ 採用** |

### なぜ案 3 なのか

**例えで言うと**: note 規約は「店内で商品を代わりに陳列してはいけない (= ブラウザ
自動化 NG)」と言ってる。案 1 は「店員に代わりに陳列したかも疑惑」、案 3 は
「商品を袋に入れて渡すだけ、陳列は客自身 (= 妻)」。疑惑の余地がない。

### 体感差の小ささ

案 1 との差は妻の**手作業 10 秒程度** (Ctrl+V + タイトル入力 + 画像ドラッグ)。
10 秒節約のためにアカウント凍結リスクを負う価値はない。

---

## Failure Modes + 対処

| 失敗モード | 影響 | 対処 |
|---|---|---|
| クリップボードが他アプリで上書きされた | 妻が貼り付けで別内容が出る | UI に「再コピー」ボタン設置。投稿履歴画面からも再取得可能 |
| 既定ブラウザが未設定 | 「note で開く」押下で何も起きない | Tauri `shell.open` の戻り値でエラー検知 → ダイアログで URL を明示表示、手動コピー可 |
| note がログアウト状態 | 投稿画面に行けない | note がログイン画面にリダイレクト、妻がログイン → 新規投稿画面へ再誘導 (アプリは関知しない) |
| 将来 note が公式 API 公開 | 手動ペースト不要になる | 新 ADR で方針切替 (案 1 相当になるが、公式 API なら規約クリア) |

---

## ロールバック手順

案 3 は「やらない」設計なので技術的にロールバック不要。
もし妻から「自動投稿したい」要望が強ければ、本 ADR を**廃案**にして新 ADR で案 1 採用。
その際は規約リスクを明記した上で妻に意思確認する。

## 影響範囲

- **ADR-001 §Revisions 2 巡目 §4**: 本 ADR の根拠 (既に記録済)
- **ADR-009 (SNS 別生成エンジン)**: note 用 `post_processor` はクリップボード用 Markdown
  生成のみ、投稿 API 呼び出しなし
- **ADR-005 (投稿履歴)**: note は `post_targets.status` 遷移に乗らない or `pending`
  のまま置く (妻の投稿完了を明示フィードバックする手段がないため、本 ADR では
  「note は履歴追跡外」とする)
- **`src/aya_afi/poster/note_clipboard.py`**: 実装は pyperclip でコピーのみ
- **Tauri `commands::open_note_compose`**: 既に Stage 0.b で実装済 (shell.open)
